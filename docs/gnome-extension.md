# GNOME Shell Extension

## Overview

The GNOME Shell extension provides the desktop user interface for Screen Time Manager.

Its sole responsibility is to display the current screen time status in the GNOME top panel. It does **not** communicate with Home Assistant, perform countdown calculations, or contain any business logic.

Instead, it acts as a lightweight view that reflects the current state produced by the Python backend.

---

# Responsibilities

The extension is responsible for:

* displaying the remaining computer time,
* showing whether the countdown is active or paused,
* presenting tooltip information,
* updating its appearance when the status changes.

The extension is **not** responsible for:

* communicating with Home Assistant,
* calculating the remaining time,
* deciding whether time should be consumed,
* locking the desktop,
* managing notifications.

Those responsibilities belong to the Python backend.

---

# Architecture

```text
                    Python Service
                          │
                          │ Atomic writes
                          ▼
              /run/user/<uid>/ha-time/
                     status.json
                          │
                          │ File monitoring
                          ▼
               GNOME Shell Extension
                          │
                          ▼
                  Top Panel Indicator
```

The extension simply watches a JSON file written by the Python service and updates the panel whenever its contents change.

This keeps the extension completely independent from the countdown logic.

---

# Status File

The extension reads a single JSON file:

```text
/run/user/<uid>/ha-time/status.json
```

The file is generated by the Python service and contains the information required to render the panel.

Example:

```json
{
  "version": 1,
  "text": "🎮 00:23:15",
  "tooltip": "Computer time remaining",
  "color": "green"
}
```

The Python service updates the file atomically, ensuring the extension never reads partially written data.

---

# Update Flow

```text
Countdown changes
        │
        ▼
Python updates status.json
        │
        ▼
GNOME extension detects change
        │
        ▼
Panel indicator is refreshed
```

The extension never calculates the countdown itself.

It simply presents the most recent state supplied by the backend.

---

# Why a Status File?

Using a small JSON file as the communication mechanism has several advantages:

* complete separation between backend and user interface,
* no direct IPC between Python and GNOME Shell,
* no dependency on Home Assistant,
* simple debugging by inspecting the file,
* easy replacement of either component.

The extension does not need to know how the values were produced—it only needs to display them.

---

# Separation of Responsibilities

## GNOME Shell Extension

Responsible for:

* monitoring the status file,
* parsing JSON,
* updating the panel,
* displaying tooltips,
* changing indicator colors.

## Python Backend

Responsible for:

* discovering desktop sessions,
* synchronizing with Home Assistant,
* calculating the countdown,
* writing the status file,
* notifications,
* enforcing screen time.

This separation keeps the extension very small while allowing the Python backend to evolve independently.

---

# Design Philosophy

The extension intentionally acts as a **view rather than a controller**.

It never makes decisions or stores application state.

Instead, it simply reflects the current state produced by the Python service.

```text
Python
    │
    └── "Here is the current status."

GNOME Extension
    │
    └── "Display it."
```

Keeping the presentation layer independent from the application logic makes both components easier to understand, test, and maintain.

---

# Future Possibilities

Because the communication between the backend and the desktop is limited to a small JSON file, the user interface can be replaced without modifying the Python service.

Possible future frontends include:

* KDE Plasma widget,
* XFCE panel plugin,
* Cinnamon applet,
* desktop notifications,
* standalone GTK application.

As long as a frontend can read the same status file format, it can display the current screen time without any knowledge of the underlying implementation.
